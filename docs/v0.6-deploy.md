# v0.6 — Azure deployment guide (Azure DevOps edition)

Stand up the portal + backend on Azure Container Apps and wire Azure DevOps
Pipelines for CI/CD. Everything runs inside Azure: compute, registry,
identity, source of truth (optional), and pipelines.

**Cost target:** ~$15–40/month at demo scale (both Container Apps scale to
zero when idle; ACR Basic is ~$5/month; Log Analytics is ingestion-based).

## Topology at a glance

```
 Azure DevOps  ──(service connection, workload identity)──►  Azure Subscription
     │                                                        │
     ├─ Repo (or connected GitHub repo)                       ├─ Resource Group
     ├─ Pipelines                                             │   ├─ ACR (Basic)
     │   ├─ ci.yml       lint + test                          │   ├─ User-Assigned MI
     │   └─ deploy.yml   build → push → roll out              │   ├─ Log Analytics
     └─ Variable group + Environment                          │   ├─ Container Apps Env
                                                              │   │   ├─ backend app
                                                              │   │   └─ portal app
                                                              │   └─ Role assignments
                                                              │       └─ AcrPull → UAMI
                                                              └─ Entra app (multi-tenant)
                                                                  ├─ FIC → UAMI principal
                                                                  └─ Graph app perms
```

## Prereqs

* Azure subscription with Contributor (or Owner) on a resource group.
* Azure DevOps organization + project (or create one at <https://dev.azure.com>).
* `az` CLI ≥ 2.60, logged in via `az login`.
* Docker Desktop only if you want to build images locally; otherwise the
  pipeline does it.

## Step 1 — Register the multi-tenant Entra app

The portal signs users in with this app; the backend validates tokens from
it; customers' Global Admins grant consent to it.

```powershell
$app = az ad app create `
  --display-name "Compliance Collector" `
  --sign-in-audience AzureADMultipleOrgs `
  --web-redirect-uris "http://localhost:3000/auth/callback" `
  | ConvertFrom-Json

$clientId = $app.appId
Write-Host "Client ID: $clientId"

# Expose the API
az ad app update --id $clientId --identifier-uris "api://$clientId"

# Add Graph delegated scopes (adjust to v0.3 manifest list)
az ad app permission add --id $clientId `
  --api 00000003-0000-0000-c000-000000000000 `
  --api-permissions `
    e1fe6dd8-ba31-4d61-89e7-88639da4683d=Scope `
    bf7b1a76-6e77-406b-b258-bf5c7720e98f=Scope
```

> The production redirect URI is added in Step 5 after we know the portal URL.

## Step 2 — Deploy the infrastructure with Bicep

```powershell
$rg     = "rg-compliance-collector"
$region = "eastus"

az group create -n $rg -l $region

az deployment group create `
  --resource-group $rg `
  --template-file infra/main.bicep `
  --parameters `
      backendImage="mcr.microsoft.com/k8se/quickstart:latest" `
      portalImage="mcr.microsoft.com/k8se/quickstart:latest" `
      entraAppClientId=$clientId `
      entraApiAudience="api://$clientId" `
      entraApiScope="api://$clientId/access_as_user"
```

Placeholder quickstart images let the apps come up before our own images
land in ACR. The pipeline swaps them out.

Capture outputs:

```powershell
$out = az deployment group show -g $rg -n main --query properties.outputs | ConvertFrom-Json

$acrName        = $out.acrName.value
$backendAppName = $out.backendAppName.value
$portalAppName  = $out.portalAppName.value
$backendUrl     = $out.backendUrl.value
$portalUrl      = $out.portalUrl.value
$uamiPrincipal  = $out.uamiPrincipalId.value
# CAE_DEFAULT_DOMAIN is everything after "<app>." in the URLs above
$caeDomain      = $backendUrl.Replace("https://$backendAppName.","")
```

## Step 3 — Wire the UAMI as a federated credential on the Entra app

Lets the backend call Graph as the app without a secret — the UAMI is the credential.

```powershell
$tenantId = az account show --query tenantId -o tsv

$fic = @{
  name        = "ccol-uami-fic"
  issuer      = "https://login.microsoftonline.com/$tenantId/v2.0"
  subject     = $uamiPrincipal
  audiences   = @("api://AzureADTokenExchange")
  description = "UAMI-backed FIC for compliance-collector backend"
} | ConvertTo-Json -Compress

$fic | Out-File fic.json -Encoding ascii
az ad app federated-credential create --id $clientId --parameters "@fic.json"
Remove-Item fic.json
```

## Step 4 — Grant Graph application permissions to the UAMI

Reuse the v0.3 admin-consent manifest. The target principal is
`$uamiPrincipal` (the UAMI's service principal object ID). Same
`Grant-MiGraphPerms.ps1` pattern you already have.

## Step 6 — Set up Azure DevOps

### 6a. Service connection (workload identity, no secrets)

1. Azure DevOps → Project settings → **Service connections** → New →
   **Azure Resource Manager** → **Workload Identity federation
   (automatic)**.
2. Scope: **Subscription** → pick your subscription → resource group
   `rg-compliance-collector`.
3. Name it `sc-ccol` → Save. This creates a federated credential on an
   app registration for you automatically.
4. Grant the service connection's app **AcrPush** on the ACR (one-time):

   ```powershell
   $sc = az devops service-endpoint list --query "[?name=='sc-ccol'] | [0]" | ConvertFrom-Json
   # If the az devops CLI isn't handy, copy the client ID from the service
   # connection page in the portal instead, then:
   az role assignment create `
     --assignee <service_connection_client_id> `
     --role AcrPush `
     --scope $(az acr show -n $acrName --query id -o tsv)
   ```

### 6b. Variable group

Pipelines → **Library** → New variable group → name
`compliance-collector-azure`. Add:

| Variable | Value |
|---|---|
| `AZURE_SUBSCRIPTION_CONNECTION` | `sc-ccol` |
| `AZURE_RESOURCE_GROUP` | `rg-compliance-collector` |
| `ACR_NAME` | `$acrName` |
| `BACKEND_APP_NAME` | `$backendAppName` |
| `PORTAL_APP_NAME` | `$portalAppName` |
| `CAE_DEFAULT_DOMAIN` | `$caeDomain` |
| `ENTRA_CLIENT_ID` | `$clientId` |
| `ENTRA_API_SCOPE` | `api://$clientId/access_as_user` |

None of these are secrets — `NEXT_PUBLIC_*` values are baked into the JS
bundle at build time, so they're public by design.

### 6c. Environment (optional approval gate)

Pipelines → **Environments** → New → name `compliance-collector-prod`.
Add yourself (or your manager) as an approver if you want a manual gate
before the Deploy stage runs.

### 6d. Register the pipelines

Pipelines → New pipeline → point at this repo (Azure Repos or GitHub) →
**Existing Azure Pipelines YAML file**:

* `/azure-pipelines/deploy.yml` — save as "Compliance Collector Deploy".
  Do not run yet.
* `/azure-pipelines/ci.yml` — save as "Compliance Collector CI". Runs on
  the next push to `main` or any PR. (Skip this one if you're keeping
  the GitHub Actions CI workflows.)

## Step 7 — First deploy

Pipelines → **Compliance Collector Deploy** → Run pipeline → Image tag
`0.6.0` → Run.

Stages:

1. **Build** — backend + portal images build in parallel, push to ACR.
2. **Deploy** — waits for environment approval (if set), then
   `az containerapp update` for both apps.

First run ~5 minutes. Subsequent runs are faster with layer caching.

Open the portal URL, sign in with a Global Admin, and you're demo-ready.

```powershell
Write-Host "Portal:  $portalUrl"
Write-Host "Backend: $backendUrl/docs"
```

## Troubleshooting

| Symptom | Fix |
|---|---|
| Portal "Failed to acquire token" | Redirect URI in Step 5 must match `$portalUrl/auth/callback` exactly (https, no trailing slash). |
| Backend 401 on portal calls | Token `aud` must match `api://<client_id>`. Check `ENTRA_API_SCOPE` in the variable group. |
| Cold start (~10s blank portal) | `minReplicas: 0` in Bicep. Bump to 1 for always-warm demos. |
| Graph call fails `AADSTS700016` | UAMI FIC (Step 3) mis-wired. `subject` must be the UAMI principal ID, not client ID. |
| `Docker@2` authorization error | Service connection doesn't have AcrPush, or variable group not linked to the pipeline. |
| Pipeline can't see variable group | In pipeline edit UI → **Variables** → Variable groups → link `compliance-collector-azure`. |

See `docs/demo-script.md` for the actual demo runbook.
See `azure-pipelines/README.md` for pipeline-specific quick reference.
